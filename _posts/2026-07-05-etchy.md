## Overview

You've spun rev B of a board, sent it to the fab, and somewhere between rev A and
rev B *something* changed on a copper layer. Which pad? How much? Did that
last-minute clearance nudge actually land, or did it quietly not? Staring at two
Gerber renders side by side is how mistakes get to production.

**etchy** points at two revisions of a board's fabrication output — Gerber,
Excellon drill, pick-and-place — and shows *and measures* exactly what changed.
It's the ground-up Rust successor to my Python tool
[Gerber-Diff-Tool](https://github.com/Cimos/Gerber-Diff-Tool), which is the thing
that actually works today (see the honesty section below). Dual-licensed MIT /
Apache-2.0.

```
etchy old/ new/     # → SVG overlay + change heatmap + JSON magnitudes
```

## Why "trustworthy" is the whole point

A diff tool that occasionally misses a change is worse than no tool, because it
teaches you to trust it and then lets you down on the one board that mattered. So
the design is built around not lying to you:

- **Same-board revisions only.** Point it at two different boards and it fails
  loud rather than emitting a confident, meaningless diff.
- **No silent misses.** The engine is backed by a golden-image corpus plus
  property and fuzz tests — the change either shows up or the run fails, never a
  quiet wrong answer.

That's the bar the predecessor set and the successor has to clear before I'll put
a green badge on it.

## One computation, three outputs

The core trick is that everything comes from a single per-layer polygon boolean
diff — `added = B − A`, `removed = A − B` — computed once and then presented three
ways:

1. A resolution-independent **SVG overlay** — the classic red/green added/removed
   view, but as vectors, so it stays crisp at any zoom.
2. A **change heatmap** — "where do I even look?", for boards where the diff is a
   handful of pads in a sea of unchanged copper.
3. **Magnitudes** — changed area and region count as JSON, so CI can gate on
   *how much* changed, not just whether anything did.

Schematic-PDF page-by-page pixel diff comes along too, carried over from the
predecessor.

## CI-first, viewer second

etchy is built to live in a pipeline first and on your desktop second: proper exit
codes, per-layer thresholds, JSON output, and PR comments, shipped as small static
binaries and a distroless container. The native [egui](https://github.com/emilk/egui)
desktop viewer is a second surface for when you want to poke around a diff by hand —
it's a separate binary that never gets compiled into the headless builds, so CI
stays lean.

## What works today, and what doesn't yet

I'd rather be straight about this than let a nice write-up imply a finished tool:

- **Works now:** the Python predecessor,
  [Gerber-Diff-Tool](https://github.com/Cimos/Gerber-Diff-Tool) v0.11. If you need
  to diff fab output today, that's the one to reach for.
- **In progress:** etchy itself — the Rust rewrite — is early. The architecture,
  crate split (`etchy-core` engine, `etchy-cli`, `etchy-gui`, `etchy-pdf`) and the
  verified library stack are settled; the engine is being built against the golden
  corpus. It is not yet a tool you should point at a real board.
- **The plan:** first usable release lands when the core diff is proven end to end
  on the corpus, at which point this post gets its "go use it" section and the
  portfolio card goes live.

## Non-goals

Scope discipline is a feature. etchy deliberately does **not** do native CAD
ingestion (KiCad/Altium/IPC-2581/ODB++), net or connectivity diff, BOM/component
diff, or DRC. It diffs fabrication output, trustworthily, and stops there — the
other jobs have other tools ([kitium](/kitium) covers a chunk of the KiCad-side CI
story).
